## 2026-06-22

- **Four local-server fixes (desktop).** Surfaced after running a second local clinic system on the same machine. Cloud node and mobile are unaffected. Full pytest suite green (+10 tests); frozen-exe boot smoke verified. Shipped via PR #17; a fresh `DentaCare-Setup.exe` (v1.1.1) was rebuilt.
  - **Replacing the database re-showed the activation popup, then re-activating hit "max devices".** The prior fix preserved the activation *settings* (`active_serial_number` / `active_serial_token` / cloud pairing) across `POST /api/data/replace`, but the license **gate** (`_license_gate_state`) decides "activated" by reading the cached row in the `licenses` table — which a replace overwrites along with the rest of the DB. With no record, the gate re-prompted, and re-activating counted a fresh device against the cloud cap. Replace now also snapshots/restores the active serial's `licenses` row and its `license_devices` rows (`_snapshot_device_license` / `_restore_device_license`), so the workstation stays activated across a data swap.
  - **Opening DentaCare could show another local app.** The service hard-coded port 5000 and the desktop window hard-coded `http://127.0.0.1:5000`, so whichever local Flask app bound 5000 first owned it — opening DentaCare next to another local system showed the *other* app. The service now uses its preferred port when free, otherwise an OS-assigned free port, and publishes the chosen port to `<data dir>/service.port`; the window reads that file (`window/service_port.py`) and always points at the right server. The session cookie was also renamed to `dentacare_session` because localhost cookies are not port-scoped, so two local apps sharing Flask's default `session` name would otherwise clobber each other's logins. Cloud node keeps its fixed Caddy port.
  - **Print Invoice failed with "you need another app to open this file".** `openPrintWindow()` used `window.open()`, which WebView2 (the packaged app) hands to the OS shell instead of opening a real popup. Reworked to print via an in-document hidden iframe (`iframe.contentWindow.print()`), which stays in-process and prints reliably in both WebView2 and a browser.
  - **Reports: the mislabeled "Lab" sub-tab is now "Custom Range" (EN/AR).** It was never a lab feature — it is a generic from/to date-range report. No behaviour change; lab-expense data is unaffected (still in the Reports "Lab Expenses" card, the Procedure Catalog's Default Lab Expense / Requires Lab, and each treatment line).

## 2026-06-20

- **Two post-activation cloud/licensing fixes (desktop).** Both surfaced right after a fresh online activation; cloud node and mobile are unaffected. Full pytest suite green (+3 regression tests).
  - **"Enable secure cloud backup" popup failed with a 403.** The post-activation popup calls `POST /api/cloud/pair` with only `{serial_number}` (no `offline_token`). `_resolve_offline_token()` looked in the request body → `app_settings['cloud_offline_token']` → env, but never `active_serial_token`. After an *online* activation only `active_serial_token` is stored (no prior pair, so `cloud_offline_token` is empty), so the pair reached the cloud's mandatory Ed25519 signed-serial gate (`CLINIC_REQUIRE_SIGNED_SERIAL=1`) unsigned and was rejected — surfaced as *"Cloud registration failed (HTTP 403)"*. `_resolve_offline_token()` now falls back to `active_serial_token` (new tier 3, before the env var), matching what `/api/cloud/enable` and the auto-pair worker already do.
  - **Replacing the database re-showed the activation popup.** `POST /api/data/replace` overwrites the whole DB with `shutil.copy2`, including the `app_settings` table where this workstation's activation (`active_serial_number` / `active_serial_token` / `device_fingerprint`) and cloud pairing live — so the swap silently de-activated the machine and the license gate re-prompted. Activation belongs to the *install*, not the *data*: replace now snapshots the device-local settings before the swap and restores them after `init_database()`, so importing/restoring data leaves the activation and cloud link intact.

## 2026-05-28

- **Standalone installer hardening — five bugs found and fixed during a full dev-machine smoke test (§3.2–§3.7 + §4).** Ran the complete install / upgrade / uninstall / migration / edge-case checklist on a real Windows 11 machine (no VM). Data preservation was verified by SHA-256 fingerprinting the DB across every uninstall/reinstall. All five fixes are in `installer\DentaCare.iss` (+ a regenerated `DentaCare.ico`); the Python suite is unaffected (193 still passing).
  - **Blurry app icon.** `DentaCare.ico` contained only a single 16×16 frame, so Windows upscaled it everywhere larger (taskbar, Start Menu, title bar, desktop shortcut) — visibly fuzzy. Regenerated as a multi-resolution icon (16/24/32/48/64/128/256 px) from the 669×740 `DentaCare.PNG`; the installed exe now embeds a true 256 px frame.
  - **Logon autostart never registered.** The "Launch DentaCare window at logon" task wrote to `HKCU\…\Run`. Under an admin (machine-wide) install, `HKCU` resolves to the elevated context, not the logged-in user's hive, so the key silently never appeared for the actual user (Inno even warned about this at compile time). Switched the autostart value to `HKLM\…\Run` — machine-wide, the correct behaviour for a single-workstation clinic, and warning-clean.
  - **Upgrade install failed with "DeleteFile failed, code 5 (access denied)".** Reinstalling over an existing install couldn't replace `DentaCare.exe`/`DentaCareService.exe` because the running service and window still held them open. The installer now stops + removes the service and closes the window at `ssInstall` (before any file is copied); the service is recreated fresh by `[Run]`. No-op on a first install.
  - **Legacy-DB migration prompt never fired.** The migration searched `{userdesktop}`/`{userdocs}` — per-user constants that, like `HKCU`, resolve to the elevated context under an admin install, so the customer's `dental_clinic.db` was never found. Rewrote it to scan the real user profiles directly (`{sd}\Users\*\{Desktop, OneDrive\Desktop, Documents}`); confirmed via the Inno install log that it now finds and copies the DB (original left in place as a backup). Added `Log()` diagnostics to the migration code.
  - **Uninstall left orphaned binaries.** `nssm.exe` + `DentaCareService.exe` remained in the install folder after uninstall because Inno's file-removal pass raced the Service Control Manager releasing the just-removed service binaries. Moved the teardown into `[Code]` at `usUninstall` and added a poll-delete retry loop (≤10 s) so the binaries are released and removed before the file pass — uninstall now removes the install folder completely.
  - **Verified working:** clean install, post-install service/log/Start-Menu state, autostart key written-on-install/removed-on-uninstall, legacy-DB migration, upgrade-over-existing, uninstall-keep-data (DB hash preserved), uninstall-delete-data, NSSM crash recovery (§4.2), and Defender Controlled Folder Access launch from Start Menu (§4.4, no `0xc0000022`).

## 2026-05-24

- **Bluetooth sync diagnostics — fix opaque "bt pairing failed" error.** First-hardware test of the BT-SPP fallback (paired Android phone ↔ Windows clinic PC) surfaced two diagnostic gaps that made the failure impossible to triage: the phone showed a generic *"bt pairing failed"* snackbar no matter what actually broke, and the desktop Settings → Bluetooth Sync card showed nothing about which phones were paired, which had connected, or whether the daemon was even listening. Both sides now surface real information.
  - **Mobile (`bluetooth_sync_service.dart`)**: `_autoPair()` previously had `catch (_) { return null; }` which swallowed every underlying exception. Replaced with a typed `_AutoPairOutcome` return type that carries the real error message — `TimeoutException` becomes *"BT connect timed out (10 s) — is the clinic PC listening?"*, generic `Exception` becomes *"BT connect failed: <toString>"*, and a rejected handshake surfaces the server's error message. `runOneSyncCycle` propagates the outcome's message into the `BtSessionResult.failure`, which the existing snackbar + status line already render.
  - **Mobile (`settings_screen.dart`)**: the persistent red error banner under the Bluetooth peer card is now a `_BtErrorCard` with `SelectableText` (so the doctor can long-press → copy the message when asking for help), wraps multi-line errors instead of overflow-ellipsis, and appends a locale-aware hint ("Tip: on the clinic PC, open Settings → Bluetooth Sync, check the COM port pill is green") when the message looks like a connect failure.
  - **Desktop (`dental_clinic.py`)**: `/api/bt/status` now returns three new fields. `server_listening` (bool) is driven by a module-level flag the daemon flips True inside the `with ser:` block and False in `finally` — so the UI can tell the user the COM port is actually open. `paired_devices` returns the `paired_devices` table (newest-first, cap 20, excludes `device_token`). `recent_attempts` returns the last 10 entries of a new in-memory `collections.deque(maxlen=20)` populated by `_bt_record_attempt(...)` from every BT op (`hello`, `bt_pair`, `sync_export`, `sync_import`), with outcome ∈ `{ok, unauthorized, error, rejected}` and a short detail. All bilingual EN/AR. Hooked into the same `/api/bt/status` fetch that already drives the pill — no extra round trip.

## 2026-05-23

- **Mobile Bluetooth sync — stability redesign.** Repeated crashes on Android 13/14 were rooted in the foreground-service stack (`flutter_background_service` + Android FGS rules): any failure in that chain killed the process with an uncatchable `RemoteServiceException` / `ForegroundServiceDidNotStartInTimeException` that no Dart `try/catch` could intercept. The fix removes the foreground-service path entirely: the 30 s BT auto-loop now runs in the main isolate and is bound to the Android activity lifecycle — it ticks while the app is on screen (or in the recent-apps cache) and pauses when the activity is `paused` / `detached`. Manual "Sync now via Bluetooth" and the BT auto-pair handshake are unchanged. **Lost capability**: silent walk-by sync when the app has been swiped from recents or the phone has rebooted — the doctor opens the app once when arriving at the clinic and sync resumes automatically. Removed dependency: `flutter_background_service` (with its platform packages). Removed permissions: `FOREGROUND_SERVICE`, `FOREGROUND_SERVICE_DATA_SYNC`, `POST_NOTIFICATIONS`.
