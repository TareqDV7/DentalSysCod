# Changelog

## 2026-05-28

- **Standalone installer hardening ‚Äî five bugs found and fixed during a full dev-machine smoke test (¬ß3.2‚Äì¬ß3.7 + ¬ß4).** Ran the complete install / upgrade / uninstall / migration / edge-case checklist on a real Windows 11 machine (no VM). Data preservation was verified by SHA-256 fingerprinting the DB across every uninstall/reinstall. All five fixes are in `installer\DentaCare.iss` (+ a regenerated `DentaCare.ico`); the Python suite is unaffected (193 still passing).
  - **Blurry app icon.** `DentaCare.ico` contained only a single 16√ó16 frame, so Windows upscaled it everywhere larger (taskbar, Start Menu, title bar, desktop shortcut) ‚Äî visibly fuzzy. Regenerated as a multi-resolution icon (16/24/32/48/64/128/256 px) from the 669√ó740 `DentaCare.PNG`; the installed exe now embeds a true 256 px frame.
  - **Logon autostart never registered.** The "Launch DentaCare window at logon" task wrote to `HKCU\‚Ä¶\Run`. Under an admin (machine-wide) install, `HKCU` resolves to the elevated context, not the logged-in user's hive, so the key silently never appeared for the actual user (Inno even warned about this at compile time). Switched the autostart value to `HKLM\‚Ä¶\Run` ‚Äî machine-wide, the correct behaviour for a single-workstation clinic, and warning-clean.
  - **Upgrade install failed with "DeleteFile failed, code 5 (access denied)".** Reinstalling over an existing install couldn't replace `DentaCare.exe`/`DentaCareService.exe` because the running service and window still held them open. The installer now stops + removes the service and closes the window at `ssInstall` (before any file is copied); the service is recreated fresh by `[Run]`. No-op on a first install.
  - **Legacy-DB migration prompt never fired.** The migration searched `{userdesktop}`/`{userdocs}` ‚Äî per-user constants that, like `HKCU`, resolve to the elevated context under an admin install, so the customer's `dental_clinic.db` was never found. Rewrote it to scan the real user profiles directly (`{sd}\Users\*\{Desktop, OneDrive\Desktop, Documents}`); confirmed via the Inno install log that it now finds and copies the DB (original left in place as a backup). Added `Log()` diagnostics to the migration code.
  - **Uninstall left orphaned binaries.** `nssm.exe` + `DentaCareService.exe` remained in the install folder after uninstall because Inno's file-removal pass raced the Service Control Manager releasing the just-removed service binaries. Moved the teardown into `[Code]` at `usUninstall` and added a poll-delete retry loop (‚â§10 s) so the binaries are released and removed before the file pass ‚Äî uninstall now removes the install folder completely.
  - **Verified working:** clean install, post-install service/log/Start-Menu state, autostart key written-on-install/removed-on-uninstall, legacy-DB migration, upgrade-over-existing, uninstall-keep-data (DB hash preserved), uninstall-delete-data, NSSM crash recovery (¬ß4.2), and Defender Controlled Folder Access launch from Start Menu (¬ß4.4, no `0xc0000022`).

  - **NSSM-supervised service.** Installer registers `DentaCare` as an auto-start service running as `LocalSystem` with log rotation at 10 MB. Crashes restart automatically (verified ¬ß4.2). Service stdout/stderr land in `C:\ProgramData\DentaCare\logs\`.

## 2026-05-24

- **Bluetooth sync diagnostics ‚Äî fix opaque "bt pairing failed" error.** First-hardware test of the BT-SPP fallback (paired Android phone ‚Üî Windows clinic PC) surfaced two diagnostic gaps that made the failure impossible to triage: the phone showed a generic *"bt pairing failed"* snackbar no matter what actually broke, and the desktop Settings ‚Üí Bluetooth Sync card showed nothing about which phones were paired, which had connected, or whether the daemon was even listening. Both sides now surface real information.
  - **Mobile (`bluetooth_sync_service.dart`)**: `_autoPair()` previously had `catch (_) { return null; }` which swallowed every underlying exception. Replaced with a typed `_AutoPairOutcome` return type that carries the real error message ‚Äî `TimeoutException` becomes *"BT connect timed out (10 s) ‚Äî is the clinic PC listening?"*, generic `Exception` becomes *"BT connect failed: <toString>"*, and a rejected handshake surfaces the server's error message. `runOneSyncCycle` propagates the outcome's message into the `BtSessionResult.failure`, which the existing snackbar + status line already render.
  - **Mobile (`settings_screen.dart`)**: the persistent red error banner under the Bluetooth peer card is now a `_BtErrorCard` with `SelectableText` (so the doctor can long-press ‚Üí copy the message when asking for help), wraps multi-line errors instead of overflow-ellipsis, and appends a locale-aware hint ("Tip: on the clinic PC, open Settings ‚Üí Bluetooth Sync, check the COM port pill is green") when the message looks like a connect failure.
  - **Desktop (`dental_clinic.py`)**: `/api/bt/status` now returns three new fields. `server_listening` (bool) is driven by a module-level flag the daemon flips True inside the `with ser:` block and False in `finally` ‚Äî so the UI can tell the user the COM port is actually open. `paired_devices` returns the `paired_devices` table (newest-first, cap 20, excludes `device_token`). `recent_attempts` returns the last 10 entries of a new in-memory `collections.deque(maxlen=20)` populated by `_bt_record_attempt(...)` from every BT op (`hello`, `bt_pair`, `sync_export`, `sync_import`), with outcome ‚àà `{ok, unauthorized, error, rejected}` and a short detail. In-memory only ‚Äî lost on restart, that's intentional.
  - **Desktop (Settings ‚Üí Bluetooth Sync card)**: under the status pill now lives a one-line "Listening on COMx ‚úì / No listener" indicator + two collapsed `<details>` blocks ‚Äî "Paired phones (N)" (name, first paired, last seen, active) and "Recent connection log" (time, device, op, outcome, detail). All bilingual EN/AR. Hooked into the same `/api/bt/status` fetch that already drives the pill ‚Äî no extra round trip.
  - **Tests**: 6 new tests in `tests/test_bt_diagnostics.py` covering empty diagnostics, `paired_devices` round-trip (with token redaction), `recent_attempts` ordering + maxlen cap, and `server_listening` flag. Total **170 passing across 22 suites** (was 164/21). Flutter side: 27 tests still pass, `flutter analyze` clean.

## 2026-05-23

- **Mobile Bluetooth sync ‚Äî stability redesign.** Repeated crashes on Android 13/14 were rooted in the foreground-service stack (`flutter_background_service` + Android FGS rules): any failure in that chain killed the process with an uncatchable `RemoteServiceException` / `ForegroundServiceDidNotStartInTimeException` that no Dart `try/catch` could intercept. The fix removes the foreground-service path entirely: the 30 s BT auto-loop now runs in the main isolate and is bound to the Android activity lifecycle ‚Äî it ticks while the app is on screen (or in the recent-apps cache) and pauses when the activity is `paused` / `detached`. Manual "Sync now via Bluetooth" and the BT auto-pair handshake are unchanged. **Lost capability**: silent walk-by sync when the app has been swiped from recents or the phone has rebooted ‚Äî the doctor opens the app once when arriving at the clinic and sync resumes automatically. Removed dependency: `flutter_background_service` (with its platform packages). Removed permissions: `FOREGROUND_SERVICE`, `FOREGROUND_SERVICE_DATA_SYNC`, `POST_NOTIFICATIONS`.
