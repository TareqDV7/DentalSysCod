# Changelog

## 2026-05-27

- **Standalone Windows desktop app ÔÇö `DentaCare-Setup.exe` installer (v1.1.0).** Converts DentaCare from a portable browser-served Flask app into a real Windows desktop application. Customers now download one `DentaCare-Setup.exe`, double-click, and after the wizard finishes they get a service that auto-starts on boot plus a Start-Menu window. The clinic data dir moves from the user's Desktop to `C:\ProgramData\DentaCare\` (so Defender Controlled Folder Access no longer blocks it ÔÇö see [Windows CFA gotcha in earlier work]), and existing portable installs are detected during install and the legacy `dental_clinic.db` is offered for one-click migration. Source-mode dev workflow (`python dental_clinic.py`) is unchanged.
  - **Two-binary split.** `DentaCareService.exe` is the headless Flask service (gated on `CLINIC_HEADLESS=1`, no browser auto-open). `DentaCare.exe` is a pywebview launcher with a native Windows window, tray icon, close-to-hide, single-instance guard via named mutex, offline-recovery page when the service is unreachable, and persistent window geometry. Both built from the same `DentaCare.spec` PyInstaller spec.
  - **NSSM-supervised service.** Installer registers `DentaCare` as an auto-start service running as `LocalSystem` with log rotation at 10 MB. Crashes restart automatically (verified ┬ž4.2). Service stdout/stderr land in `C:\ProgramData\DentaCare\logs\`.
  - **Inno Setup 6 installer.** `installer\DentaCare.iss` produces a ~57 MB `DentaCare-Setup.exe` that installs WebView2 Evergreen if missing, provisions a Bluetooth Incoming SPP COM port via `installer\provision_bt.ps1` (falls back to opening the Bluetooth COM Ports dialog), registers the service, migrates legacy DBs, and on uninstall preserves `C:\ProgramData\DentaCare\` behind a default-No prompt.
  - **Smoke-test fixes shipped during validation:**
    - `667f55c` Offline-page recovery is now driven from Python (the in-page JS poll couldn't recover because WebView2 blocks `fetch()` from a `file://` page to an `http://` origin). The launcher polls health from Python and reloads the window directly.
    - `a844f0c` `register-service.bat` / `unregister-service.bat` self-elevate via UAC and pause on exit so customers can read errors instead of the cmd window vanishing.
    - `c5913ba` `window-state.json` no longer persists the Windows hidden-window sentinel (`-32000, -32000, 160├Ś28`) ÔÇö close-to-hide used to leave that geometry on disk and the next launch would open invisible. New `is_hidden_geometry()` helper with 6 unit tests.
    - `d0b60b8` Inno Setup compile is warning-clean (`RunOnceId` for the BT provisioning step, `FileCopy` Ôćĺ `CopyFile` for the legacy DB migration).
  - **Tests**: 193 passing across 26 suites (was 187 ÔÇö added `test_window_state.py` for `is_hidden_geometry()`).

## 2026-05-24

- **Bluetooth sync diagnostics ÔÇö fix opaque "bt pairing failed" error.** First-hardware test of the BT-SPP fallback (paired Android phone Ôćö Windows clinic PC) surfaced two diagnostic gaps that made the failure impossible to triage: the phone showed a generic *"bt pairing failed"* snackbar no matter what actually broke, and the desktop Settings Ôćĺ Bluetooth Sync card showed nothing about which phones were paired, which had connected, or whether the daemon was even listening. Both sides now surface real information.
  - **Mobile (`bluetooth_sync_service.dart`)**: `_autoPair()` previously had `catch (_) { return null; }` which swallowed every underlying exception. Replaced with a typed `_AutoPairOutcome` return type that carries the real error message ÔÇö `TimeoutException` becomes *"BT connect timed out (10 s) ÔÇö is the clinic PC listening?"*, generic `Exception` becomes *"BT connect failed: <toString>"*, and a rejected handshake surfaces the server's error message. `runOneSyncCycle` propagates the outcome's message into the `BtSessionResult.failure`, which the existing snackbar + status line already render.
  - **Mobile (`settings_screen.dart`)**: the persistent red error banner under the Bluetooth peer card is now a `_BtErrorCard` with `SelectableText` (so the doctor can long-press Ôćĺ copy the message when asking for help), wraps multi-line errors instead of overflow-ellipsis, and appends a locale-aware hint ("Tip: on the clinic PC, open Settings Ôćĺ Bluetooth Sync, check the COM port pill is green") when the message looks like a connect failure.
  - **Desktop (`dental_clinic.py`)**: `/api/bt/status` now returns three new fields. `server_listening` (bool) is driven by a module-level flag the daemon flips True inside the `with ser:` block and False in `finally` ÔÇö so the UI can tell the user the COM port is actually open. `paired_devices` returns the `paired_devices` table (newest-first, cap 20, excludes `device_token`). `recent_attempts` returns the last 10 entries of a new in-memory `collections.deque(maxlen=20)` populated by `_bt_record_attempt(...)` from every BT op (`hello`, `bt_pair`, `sync_export`, `sync_import`), with outcome Ôłł `{ok, unauthorized, error, rejected}` and a short detail. In-memory only ÔÇö lost on restart, that's intentional.
  - **Desktop (Settings Ôćĺ Bluetooth Sync card)**: under the status pill now lives a one-line "Listening on COMx Ôťô / No listener" indicator + two collapsed `<details>` blocks ÔÇö "Paired phones (N)" (name, first paired, last seen, active) and "Recent connection log" (time, device, op, outcome, detail). All bilingual EN/AR. Hooked into the same `/api/bt/status` fetch that already drives the pill ÔÇö no extra round trip.
  - **Tests**: 6 new tests in `tests/test_bt_diagnostics.py` covering empty diagnostics, `paired_devices` round-trip (with token redaction), `recent_attempts` ordering + maxlen cap, and `server_listening` flag. Total **170 passing across 22 suites** (was 164/21). Flutter side: 27 tests still pass, `flutter analyze` clean.

## 2026-05-23

- **Mobile Bluetooth sync ÔÇö stability redesign.** Repeated crashes on Android 13/14 were rooted in the foreground-service stack (`flutter_background_service` + Android FGS rules): any failure in that chain killed the process with an uncatchable `RemoteServiceException` / `ForegroundServiceDidNotStartInTimeException` that no Dart `try/catch` could intercept. The fix removes the foreground-service path entirely: the 30 s BT auto-loop now runs in the main isolate and is bound to the Android activity lifecycle ÔÇö it ticks while the app is on screen (or in the recent-apps cache) and pauses when the activity is `paused` / `detached`. Manual "Sync now via Bluetooth" and the BT auto-pair handshake are unchanged. **Lost capability**: silent walk-by sync when the app has been swiped from recents or the phone has rebooted ÔÇö the doctor opens the app once when arriving at the clinic and sync resumes automatically. Removed dependency: `flutter_background_service` (with its platform packages). Removed permissions: `FOREGROUND_SERVICE`, `FOREGROUND_SERVICE_DATA_SYNC`, `POST_NOTIFICATIONS`.
